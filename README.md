  <p>The artifact I used for software engineering and design is my InventoryTracker Android app. This was originally created as my final project for CS-360: Mobile Architecture and Programming in my previous term. The app is designed for warehouse item management, where users can log in/create account, view a grid of all their inventory items, and add/update/delete items. It also has other functionality including a search bar to search for items by name and SMS notifications for low stock items. It is written in Java with an MVVM architecture and originally used SQLite with Room for data storage and persistence.
I chose to use this artifact for my ePortfolio for several reasons. The first reason is that I enjoyed working on this app and I am proud of how it turned out. Another reason is that it was one of the most complex projects I have worked on, having authentication, data persistence, MVVM architecture, and notifications. Because of this, it demonstrates a wide range of software engineering and design skills. The final reason goes along with the previous reason I gave, which is that it is a great candidate for enhancements that make it more production-level with more real-world value. After submitting the project, we had a discussion post where we talked about putting our apps on the Play Store, and I explained how I would like to migrate to a cloud database and make some other improvements before publishing, and this ePortfolio is the perfect opportunity to make those changes.
The artifact before enhancements demonstrates my development skills with user authentication, data storage and persistence, thorough design with an MVVM architecture, deploying notifications, documentation with extensive code comments, and ability to meet requirements for a software project.
The artifact has been improved significantly with my enhancements for this category. For starters, I made a big architectural change to use a single activity, multi-fragment architecture instead of separate activities for the four screens of the app. This change made the app more lightweight, removed redundant boilerplate activity code, and reduced my dependency on deprecated APIs for sharing data between activities because it is simpler with fragments. Another big change is using Hilt for dependency injection. This simplified handling dependencies and singletons and also makes the code easier to maintain and test because ViewModels can be constructed with mock repositories in tests. Next is splitting the monolithic InventoryRepository into UserRepository and ItemRepository files to separate concerns between operations for authentication and item CRUD operations. Next is authentication, which has migrated from comparing plaintext credentials from the local User table to Firebase Authentication. This significantly improved security and simplified secure credential storage and token management. Along with this change, I added session persistence such that you stay logged in after closing and re-opening the app. I also used Firebase Firestore to migrate data storage to the cloud. Firebase Authentication removed the need for the Users table, but the Items table as well as the new QuantityHistory and AuditLog tables are now stored in the cloud, allowing data access across different devices. Firestore also supports local offline caching, so the app is still usable offline and automatically syncs changes when it goes back online. Along with this change, I also added data isolation between users. The app previously showed all items in the database to all users, but with the migration to Firestore, I restructured the data layer to scope Item collections to user IDs so users only see their items. The new QuantityHistory table logs quantity changes for items, which I have used to add a line chart to the item details view to display quantity changes over time. The new AuditLog table is for more general logging used for an accountability trail, tracking all changes including add, update, and delete operations, including the ID of the user who made the change and a timestamp of the change. I also made some efficiency updates with the RecyclerView by replacing notifyDataSetChanged(), which redraws the entire items grid on every change, with ListAdapter and DiffUtil, which calculate the minimum set of changes to only redraw what is necessary. Another improvement was replacing dummy SMS notifications with real push notifications that deploy when an item’s quantity reaches 0. As a part of this change I extracted the notification methods to a separate helper utility which cleaned up the code and made it possible to send notifications from any fragment, which wasn’t possible before. The last major addition is testing. I added comprehensive unit tests for both ViewModels using Junit and Mockito. The codebase already had good comments, but I also improved the depth of the comments and added comments for all the new code, improving documentation. Additionally, I made lots of small improvements with new helper utilities, more defensive checks, and fixed other small issues I mentioned in my code review like adding default cases for all switch statements.
These enhancements certainly met the course outcomes I planned to achieve for this milestone. I identified that I planned to meet outcomes 2, 3, and 4 with this category of enhancements. I met outcome 2 for professional-quality communication through my code review and this narrative, but also largely through my code comments, which are extremely thorough, describing every method and essentially every logic branch and anything else that could be ambiguous. I met outcome 3 for designing and evaluating computing solutions using algorithmic principles and managing trade-offs through the use of things like ListAdapter and DiffUtil to improve the RecyclerView efficiency, using Firestore for storage and persistence with a locally synced cache, and migrating to a multi-fragment architecture to better suit my needs and make the app more lightweight. For outcome 4 demonstrating innovative techniques, skills, and tools, I used things like Firebase Firestore for storage and cloud sync, Firebase Authentication, Hilt dependency injection, extracting repeated or shared code to utility classes, and refactoring to use more modern and efficient APIs like ListAdapter and DiffUtil.
As far as updates to my outcome-coverage plans, I have noticed that a lot of the enhancements I was planning to make with this artifact for the databases category were intertwined with my enhancements for this milestone. For databases, my planned enhancements were Firebase Authentication, local encryption with SQLCipher, user data isolation, database indexing, and audit logging. With this enhancement, I have already implemented Firebase Auth, user data isolation, and audit logging. Additionally, since I switched out SQLite and Room for firebase’s native local sync, I can no longer use SQLCipher for local encryption, and from my research I don’t believe there is any way to encrypt Firebase’s local cache. The only thing left is database indexing, which is also not very straightforward because Firebase automatically creates indexes for every field. All I could do is add indexes for composite queries on multiple fields. I do have ideas for new enhancements for databases like scoping data isolation to users within an organization instead of just per-user, adding more rules to the Firestore database, and I have a planned enhancement to move search filtering to the database layer from this milestone that I will do for databases instead. My planned outcomes for databases are 4 and 5 for innovative techniques and security mindset, and I believe I could still demonstrate these outcomes with the database-related enhancements I already made like the cloud migration, authentication, repository separation, data isolation, and new log tables as well as those new enhancements I mentioned.
During my enhancement of the artifact, I learned a lot and faced several challenges. Migrating to Firebase for Firestore and Authentication was a significant learning experience and I faced several challenges along the way. The initial setup wasn’t too bad since there is a lot of documentation, but things like snapshot listeners and offline persistence were very different from their equivalents with SQLite and Room. Room was more simple with asynchronous calls managed with an ExecutorService. Firestore is more callback driven and manages its own threading, which was tricky to understand and work with. I also struggled with Firebase Authentication because there wasn’t great documentation, at least that I could find, for the error codes returned by Firebase for authentication. I had to manually test every situation and log the error codes to set up my error handling properly. Another issue I had with this was configuration settings. I knew that there was an error code for an existing email but incorrect password, and it turned out that it only throws that code when email enumeration is turned off, which is a feature that prevents probing for registered emails by not identifying whether an email is registered or not.
Another challenge I faced was debugging a problem where the increase and decrease buttons on the item cards to increment/decrement quantity would only visually update every other time I clicked them. It took me a while to figure out, and it ended up being a problem with real-time updates with DiffUtil, which compares an item before and after an update to determine if it should be redrawn in the UI. Since I was updating the item in place, like adding or subtracting 1 from it, then calling update with that item, DiffUtil was comparing the item I updated in place with the “new” item. Because I updated it in place, the “old” item was already updated, so it thought they were the same and did not redraw the item. Once I traced the whole data flow, I realized all I had to do was create a new item to update the quantity and run the update on that item.
The process of adding Hilt was also a learning experience for me. It took a lot of research and running through the data flow to understand how Hilt actually works, and how to properly use it.
