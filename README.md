  <p>The artifact I used for software engineering and design is my InventoryTracker Android app. This was originally created as my final project for CS-360: Mobile Architecture and Programming in my previous term. The app is designed for warehouse item management, where users can log in/create account, view a grid of all their inventory items, and add/update/delete items. It also has other functionality including a search bar to search for items by name and SMS notifications for low stock items. It is written in Java with an MVVM architecture and originally used SQLite with Room for data storage and persistence.</p>
    
<p>I chose to use this artifact for my ePortfolio for several reasons. The first reason is that I enjoyed working on this app and I am proud of how it turned out. Another reason is that it was one of the most complex projects I have worked on, having authentication, data persistence, MVVM architecture, and notifications. Because of this, it demonstrates a wide range of software engineering and design skills. The final reason goes along with the previous reason I gave, which is that it is a great candidate for enhancements that make it more production-level with more real-world value. After submitting the project, we had a discussion post where we talked about putting our apps on the Play Store, and I explained how I would like to migrate to a cloud database and make some other improvements before publishing, and this ePortfolio is the perfect opportunity to make those changes.</p>

<p>The artifact before enhancements demonstrates my development skills with user authentication, data storage and persistence, thorough design with an MVVM architecture, deploying notifications, documentation with extensive code comments, and ability to meet requirements for a software project.</p>

<p>The artifact has been improved significantly with my enhancements for this category. For starters, I made a big architectural change to use a single activity, multi-fragment architecture instead of separate activities for the four screens of the app. This change made the app more lightweight, removed redundant boilerplate activity code, and reduced my dependency on deprecated APIs for sharing data between activities because it is simpler with fragments. Another big change is using Hilt for dependency injection. This simplified handling dependencies and singletons and also makes the code easier to maintain and test because ViewModels can be constructed with mock repositories in tests. Next is splitting the monolithic InventoryRepository into UserRepository and ItemRepository files to separate concerns between operations for authentication and item CRUD operations. Next is authentication, which has migrated from comparing plaintext credentials from the local User table to Firebase Authentication. This significantly improved security and simplified secure credential storage and token management. Along with this change, I added session persistence such that you stay logged in after closing and re-opening the app. I also used Firebase Firestore to migrate data storage to the cloud. Firebase Authentication removed the need for the Users table, but the Items table as well as the new QuantityHistory and AuditLog tables are now stored in the cloud, allowing data access across different devices. Firestore also supports local offline caching, so the app is still usable offline and automatically syncs changes when it goes back online. Along with this change, I also added data isolation between users. The app previously showed all items in the database to all users, but with the migration to Firestore, I restructured the data layer to scope Item collections to user IDs so users only see their items. The new QuantityHistory table logs quantity changes for items, which I have used to add a line chart to the item details view to display quantity changes over time. The new AuditLog table is for more general logging used for an accountability trail, tracking all changes including add, update, and delete operations, including the ID of the user who made the change and a timestamp of the change. I also made some efficiency updates with the RecyclerView by replacing notifyDataSetChanged(), which redraws the entire items grid on every change, with ListAdapter and DiffUtil, which calculate the minimum set of changes to only redraw what is necessary. Another improvement was replacing dummy SMS notifications with real push notifications that deploy when an item’s quantity reaches 0. As a part of this change I extracted the notification methods to a separate helper utility which cleaned up the code and made it possible to send notifications from any fragment, which wasn’t possible before. The last major addition is testing. I added comprehensive unit tests for both ViewModels using Junit and Mockito. The codebase already had good comments, but I also improved the depth of the comments and added comments for all the new code, improving documentation. Additionally, I made lots of small improvements with new helper utilities, more defensive checks, and fixed other small issues I mentioned in my code review like adding default cases for all switch statements.</p>

<p>During my enhancement of the artifact, I learned a lot and faced several challenges. Migrating to Firebase for Firestore and Authentication was a significant learning experience and I faced several challenges along the way. The initial setup wasn’t too bad since there is a lot of documentation, but things like snapshot listeners and offline persistence were very different from their equivalents with SQLite and Room. Room was more simple with asynchronous calls managed with an ExecutorService. Firestore is more callback driven and manages its own threading, which was tricky to understand and work with. I also struggled with Firebase Authentication because there wasn’t great documentation, at least that I could find, for the error codes returned by Firebase for authentication. I had to manually test every situation and log the error codes to set up my error handling properly. Another issue I had with this was configuration settings. I knew that there was an error code for an existing email but incorrect password, and it turned out that it only throws that code when email enumeration is turned off, which is a feature that prevents probing for registered emails by not identifying whether an email is registered or not.</p>

<p>Another challenge I faced was debugging a problem where the increase and decrease buttons on the item cards to increment/decrement quantity would only visually update every other time I clicked them. It took me a while to figure out, and it ended up being a problem with real-time updates with DiffUtil, which compares an item before and after an update to determine if it should be redrawn in the UI. Since I was updating the item in place, like adding or subtracting 1 from it, then calling update with that item, DiffUtil was comparing the item I updated in place with the “new” item. Because I updated it in place, the “old” item was already updated, so it thought they were the same and did not redraw the item. Once I traced the whole data flow, I realized all I had to do was create a new item to update the quantity and run the update on that item.</p>

<p>The process of adding Hilt was also a learning experience for me. It took a lot of research and running through the data flow to understand how Hilt actually works, and how to properly use it.

## Data Structures & Algorithms Enhancement:
<a href="https://github.com/benc4/CornerGrocer/tree/category-2-enhancements" target="_blank">https://github.com/benc4/CornerGrocer/tree/category-2-enhancements</a>

<details>
  <summary>Narrative</summary>
  <p>The artifact I chose for my data structures and algorithms enhancement is my Corner Grocer C++ program. It reads a text file listing every item sold in a grocery store over the course of a day and counts the frequency of each item to display information about their frequencies through a menu interface. It was originally created in April 2025 for my CS 210 Programming Languages course.</p>
    
<p>I chose to include this artifact in my ePortfolio because it is a perfect candidate for data structure and algorithm enhancements due to it being centered around operations on a data structure, which was originally a std::map. I originally identified that the use of the std::map was inefficient because the sorted order of it was never utilized, and from there I saw more improvements I could make to the program using different data structures and algorithms.</p>
  
<p>The first change I made was replacing the std::map with std::unordered_map. This change made the program more efficient because std::map is an ordered and balanced binary search tree with O(log n) insertion and lookup times, and since the program never used the sorted order, the switch to std::unordered_map brought those times down to O(1) by hashing each item name directly into a bucket instead. The next change was adding a trie data structure to improve searching for items and enabled the autocomplete functionality with prefix searching. The trie stores each item name character by character and keeps the frequency count at the node at the end of each word. Walking the trie to a prefix is O(m) in the length of the prefix and is independent of how many items are stored. Autocomplete works by using a depth-first search to find every complete word beneath the prefix, which is much faster than the equivalent feature with a hash map. Next was adding a top-k frequency search feature, which returns the top k most purchased items. It uses a max-heap algorithm with a std::priority_queue to store each item and frequency count and then pop the top k from it. Building the heap is O(n) and each of the k pops are O(log n), meaning the whole operation is O(n + K log n). This is more efficient than fully sorting the data when the k value is small, which is the typical use case for the feature (ex. finding the top 5 out of 50 items). Lastly, I added a benchmarking option to the menu which runs insert, lookup, and prefix search for each data structure and top-k retrieval for both algorithms, logs the time for each operation, and compares those times to the theoretical Big-O complexities.</p>
  
<p>These enhancements improved the artifact by improving efficiency, adding new features with efficient data structures and algorithms, and presenting the trade-offs between the different data structures and algorithms with benchmarking to show the theoretical differences along with the actual measured differences.</p>

<p>The course outcomes I planned to meet with these enhancements were outcomes 1, 3, and 4. Outcome 1 for building collaborative environments was met through this enhancement with the benchmarking comparison table, which makes complex data structures and algorithmic tradeoffs understandable to technical and non-technical audiences. I met outcome 3 for solving problems with algorithmic principles and computer science practices through the change from std::map to std::unordered_map, the inclusion of data structures and algorithms for new features, and the justification of these changes through the benchmarking feature which shows the tradeoffs for each. Lastly, I met outcome 4 for using innovative techniques, skills, and tools to deliver value through the data structures and algorithms that power the new value-adding features. The trie enables autocomplete prefix searching and the max-heap allows for efficient top-k retrieval.</p>
  
<p>There are no updates to the coverage plans for this artifact because everything I planned was covered with this enhancement.</p>
  
<p>Throughout the process of this enhancement, the biggest thing I learned was that each choice had tradeoffs that may be improve one type of operation, but slow down another. For example, switching from std::map to std::unordered_map brought average lookups down to O(1), but got rid of the sorted order of the map that could be useful for other operations. The trie makes prefix search independent of the size of the dataset, but it requires a separate node for each character, which trades memory usage for efficiency. For the top-k search, the max-heap is efficient while k is small, but is actually slower than a full sort for larger k values. Working through these tradeoffs helped me learn a lot about the pros and cons of different data structures and algorithms and helped me develop skills in analyzing tradeoffs between them for different applications.</p>
  
<p>A challenge I faced was managing memory with the trie data structure. To avoid memory leaks, I had to write the destructor to handle cleanup for each node. I couldn’t use automatic cleanup or just destroy the root node because each node is created with a raw pointer, meaning removing a parent node would break the connection to child nodes and cause a leak. After some iterating and research, I was able to develop a recursive destructor that clears nodes from the leaves upward and verifies nothing is left before completing.
  </p>
